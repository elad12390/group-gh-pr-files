# Contributing to PR File Grouper for GitHub

Thanks for your interest in improving this extension! It's a cross-browser
(Firefox + Chrome) MV3 extension with a small, dependency-free runtime and a
real-browser end-to-end test suite. Contributions of all kinds are welcome —
bug reports, feature ideas, docs, and code.

By participating, you agree to abide by our [Code of Conduct](CODE_OF_CONDUCT.md).

## Ways to contribute

- **Report a bug** — open a [bug report](https://github.com/elad12390/group-gh-pr-files/issues/new?template=bug_report.yml).
- **Request a feature** — open a [feature request](https://github.com/elad12390/group-gh-pr-files/issues/new?template=feature_request.yml).
- **Send a pull request** — see the workflow below.

## Prerequisites

- [Bun](https://bun.sh) (preferred) — `curl -fsSL https://bun.sh/install | bash`
- **Firefox** + [`geckodriver`](https://github.com/mozilla/geckodriver) for the
  Firefox E2E run — `brew install geckodriver`
- **Chromium** (via Playwright) for the Chrome E2E run —
  `bunx playwright install chromium`

## Getting started

```bash
git clone https://github.com/elad12390/group-gh-pr-files.git
cd group-gh-pr-files
bun install
```

Load the extension unpacked while you work:

- **Firefox** — `about:debugging#/runtime/this-firefox` → *Load Temporary
  Add-on…* → pick `manifest.json`. Or run a scratch browser with
  `bunx web-ext run --source-dir .`.
- **Chrome** — `chrome://extensions` → enable *Developer mode* → *Load unpacked*
  → pick this folder.

## Project layout

| Path | What |
| --- | --- |
| `manifest.json` | Single MV3 manifest for both browsers |
| `src/content.js` | All runtime logic + the shadow-DOM panel |
| `src/popup.html` / `src/popup.js` | Toolbar button that toggles the panel |
| `icons/` | `icon.svg` source; PNGs are generated by `bun run icons` |
| `.idd/` | Intent spec — **what** the extension must do |
| `.bdd/` | Gherkin features + real Firefox/Chrome E2E runners + fixtures |

## Dev scripts

```bash
bun run lint             # web-ext lint — must be 0 errors, 0 warnings
bun run build            # package to web-ext-artifacts/*.zip
bun run icons            # regenerate PNG icons from icons/icon.svg
bun run test:e2e:firefox # real Firefox via geckodriver
bun run test:e2e:chrome  # real Chrome via Playwright (--load-extension)
```

## Coding conventions

- **No build step, no framework, no runtime dependencies.** The content script
  is plain ES that runs directly in the page. Keep it that way.
- **Match the existing style** in `src/content.js` — it's the source of truth
  for naming, structure, and DOM patterns.
- The panel lives in a **shadow root**; never leak styles into the page or rely
  on GitHub's classes staying stable. Diff hide/star is driven by a single
  page-level `<style>` keyed by each file's stable diff id.
- Support **both** the classic `Files changed` tab and the new `changes`
  experience, and **both** browsers (a `browser`/`chrome` shim is already in
  place).
- **Never** commit secrets, tokens, or machine-specific absolute paths.

## Definition of done (Bulletproof)

This project follows an intent-first, real-system-tested workflow. A behavior
change is not done until:

1. **Intent** is updated — add/adjust the constraint or example in
   [`.idd/modules/pr-file-grouper/INTENT.md`](.idd/modules/pr-file-grouper/INTENT.md).
2. **Spec** is updated — reflect it in
   [`.bdd/features/pr-file-grouper.feature`](.bdd/features/pr-file-grouper.feature).
3. **E2E proves it** — extend the real-browser runners in `.bdd/steps/` so the
   behavior is verified in a real browser (no mocks).
4. `bun run lint` is clean and **both** E2E suites pass locally.

## Pull request workflow

1. Fork and create a topic branch off `main`.
2. Make your change; keep it focused (one logical change per PR).
3. Run `bun run lint`, `bun run build`, and the E2E suites.
4. Update `README.md` / `CHANGELOG.md` if behavior or usage changed.
5. Open the PR using the template and fill in the checklist.

CI runs `web-ext lint` and `build` on every pull request; the browser E2E
suites are run locally (they need a real Firefox/Chromium).

## Reporting security issues

Please do **not** open public issues for vulnerabilities — see
[SECURITY.md](SECURITY.md).
